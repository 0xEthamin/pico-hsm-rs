# ATECC608B configuration zone layout for mini-HSM

This document is the authoritative specification of the 128-byte configuration
zone that the mini-HSM firmware writes to a blank ATECC608B-SSHDA-T chip
during provisioning. Once the configuration zone is locked, **none of these
values can ever be changed** for the life of the chip.

Reading this document is enough to write a correct configuration generator
without any other context. Every value is justified by its source. No magic
numbers.

## Status

The configuration zone written by `tools/config-generator/` follows this
specification exactly. The bytes that fall into the read-only factory area
(bytes 0 to 15) are not transmitted to the chip; they are present in the
generator output only for reference.

The expected CRC-16 of the writable portion (bytes 16 to 127) of the final
blob is **`0xC92D`**. Any deviation from this value indicates a mismatch
between the generated bytes and this specification.

## Sources of truth

All field definitions come from the Microchip CryptoAuthLib repository,
`lib/calib/calib_device.h`, in the `atecc608_config_t` structure and the
`ATCA_*_SHIFT` / `ATCA_*_MASK` constants. CryptoAuthLib is public open
source under a Microchip license. The ATECC608B datasheet itself is under
NDA, but every byte position and every bit field in this specification is
recoverable from the public CryptoAuthLib source code and matches its
working configurations.

## Layout overview

The configuration zone is 128 bytes, structured as follows.

| Offset | Size | Field                  | Writable before lock? | Notes                                            |
|--------|------|------------------------|-----------------------|--------------------------------------------------|
| 0-3    | 4    | SN03                   | No (factory)          | Serial number bytes 0-3.                         |
| 4-7    | 4    | RevNum                 | No (factory)          | Chip revision.                                   |
| 8-11   | 4    | SN47                   | No (factory)          | Serial number bytes 4-7.                         |
| 12     | 1    | SN8                    | No (factory)          | Serial number byte 8 (always 0xEE).              |
| 13     | 1    | AES_Enable             | No (factory)          | AES support flag, factory-set.                   |
| 14     | 1    | I2C_Enable             | No (factory)          | I2C support flag, factory-set.                   |
| 15     | 1    | Reserved1              | No (factory)          | Reserved, factory-set.                           |
| 16     | 1    | I2C_Address            | Yes                   | Chip 7-bit I2C address in 8-bit form.            |
| 17     | 1    | Reserved2              | Yes (must stay 0x00)  | Required to be 0x00.                             |
| 18     | 1    | CountMatch             | Yes                   | Counter match feature configuration.             |
| 19     | 1    | ChipMode               | Yes                   | Clock divider, watchdog length, TTL mode.        |
| 20-51  | 32   | SlotConfig[16]         | Yes                   | Per-slot read/write policy (2 bytes per slot).   |
| 52-59  | 8    | Counter0               | Yes                   | Initial value for monotonic counter 0.           |
| 60-67  | 8    | Counter1               | Yes                   | Initial value for monotonic counter 1.           |
| 68     | 1    | UseLock                | Yes                   | Enables UseLock feature, not used here.          |
| 69     | 1    | VolatileKeyPermission  | Yes                   | Volatile key feature, not used here.             |
| 70-71  | 2    | SecureBoot             | Yes                   | Secure boot configuration, not used here.        |
| 72     | 1    | KdflvLoc               | Yes                   | KDF input vector location.                       |
| 73-74  | 2    | KdflvStr               | Yes                   | KDF input vector seed.                           |
| 75-83  | 9    | Reserved3              | Yes (must stay 0x00)  | Reserved bytes.                                  |
| 84     | 1    | UserExtra              | Yes                   | One-time write field, kept at 0x00.              |
| 85     | 1    | UserExtraAdd (Selector)| Yes                   | One-time write field, kept at 0x00.              |
| 86     | 1    | LockValue              | Yes (until data lock) | 0x55 = data zone unlocked.                       |
| 87     | 1    | LockConfig             | Yes (until config lock)| 0x55 = config zone unlocked.                    |
| 88-89  | 2    | SlotLocked             | Yes                   | Per-slot lock bitmap, 1=unlocked.                |
| 90-91  | 2    | ChipOptions            | Yes                   | POST, IO protection, KDF/AES, etc.               |
| 92-95  | 4    | X509format             | Yes                   | X.509 signature formatting, not used here.       |
| 96-127 | 32   | KeyConfig[16]          | Yes                   | Per-slot key policy (2 bytes per slot).          |

## Reading the factory area (bytes 0-15)

The first 16 bytes of the configuration zone are written at the factory by
Microchip and are read-only for the entire life of the chip. They are
nonetheless readable in cleartext at any time, even before any provisioning
or any lock has happened. Two commands are available.

`Info(mode = Revision)` returns 4 bytes corresponding to bytes 4-7 of the
configuration zone, that is, the `RevNum` field. The expected value for
an ATECC608B-M0 is `00 00 60 02`, for M1 it is `00 00 60 03`, for M2 it
is `00 00 60 04`. This is the fastest way to verify that the chip
responding on the I2C bus is a genuine ATECC608B.

`Read(zone = Config, block = 0)` returns the 32 bytes at offsets 0-31.
This yields the full factory area in one transaction. The 9-byte chip
serial number is assembled from the result:

```
serial[0..4]  = config_bytes[0..4]   (SN03 field)
serial[4..8]  = config_bytes[8..12]  (SN47 field)
serial[8]     = config_bytes[12]     (SN8 field, always 0xEE on ATECC608)
```

This serial number is unique per physical chip. The mini-HSM firmware
uses it as a per-chip salt mixed into the `CheckMac` MAC computation
(via the `other_data` field) so that a successful CheckMac response on
one chip cannot be replayed against another chip carrying the same slot
value. It is also used to identify a token across reboots without
storing any state in the chip's data zone.

The slot 8 I/O Protection master key is **not** derived from the
serial. It is pure random output from the chip's RNG at provisioning
time. See `## Slot 8: I/O Protection master key` below for details.

The bring-up procedure on a fresh chip is therefore:

1. Wake the chip via the GPIO pulse on SDA. Confirm the wake response
   `04 11 33 43`.
2. Issue `Info(Revision)`. Confirm the response starts with `00 00 60`.
3. Issue `Read(Config, block 0)`. Extract the serial number and record it
   alongside the test bench. Confirm byte 12 (`SN8`) is `0xEE`.

If all three checks pass, the chip is genuine, the I2C bus is functional,
and the wake and polling logic in the firmware works. None of these
operations modify the chip in any way. The chip remains in its pristine,
fully-reprovisioning-capable state.

## Field encoding and chosen values

This section describes how each writable byte is encoded and what value the
mini-HSM uses. Each bit is justified.

### Byte 16: I2C_Address

The chip stores the I2C address in the 8-bit form (the 7-bit address shifted
left by one bit). The default 7-bit address is `0x60`, which yields `0xC0`
in the 8-bit form.

```
Value = 0xC0
```

### Byte 17: Reserved2

Per the ATECC608B reference, this byte must be left at `0x00`. Writing
anything else is undefined behaviour.

```
Value = 0x00
```

### Byte 18: CountMatch

The CountMatch feature is unused here, kept disabled.

```
Value = 0x00
```

### Byte 19: ChipMode

Encoded as follows:

| Bit  | Field           | Mask   | Meaning                                                |
|------|-----------------|--------|--------------------------------------------------------|
| 0    | I2C_Extra       | 0x01   | Extra I2C address feature (kept disabled).             |
| 1    | TTL_Enable      | 0x02   | TTL input level (kept disabled, fixed Vcc reference).  |
| 2    | Watchdog_Long   | 0x04   | Watchdog duration: 0 = short (~1.3s), 1 = long (~13s). |
| 3-7  | Clock_Divider   | 0xF8   | Clock divider mode: 0 = M0, 0x05 = M1, 0x0D = M2.      |

The mini-HSM chooses M0 (maximum speed, USB-powered) and the short
watchdog. All other bits at 0.

```
Value = 0x00
```

If the project transitions to battery operation later, M2 (`Clock_Divider = 0x0D`)
will reduce active current to about 3 mA at the cost of roughly 4x slower
operations. A change to ChipMode requires writing a new configuration zone
to a fresh chip; locked chips cannot be modified.

### Bytes 20-51: SlotConfig[16]

Each SlotConfig is 2 bytes, encoded as a little-endian u16 word. The
semantics of each bit depend on whether the slot holds an ECC private key
(`KeyConfig.Private = 1`) or generic data.

#### SlotConfig layout for ECC private key slots

| Bit   | Field          | Mask     | Meaning                                                |
|-------|----------------|----------|--------------------------------------------------------|
| 0     | ExtSig         | 0x0001   | Allow external-message ECDSA Sign on this slot.        |
| 1     | IntSig         | 0x0002   | Allow internal-message ECDSA Sign.                     |
| 2     | ECDH           | 0x0004   | Allow ECDH on this slot.                               |
| 3     | WriteECDH      | 0x0008   | Allow writing the output of ECDH back into the slot.   |
| 4     | NoMac          | 0x0010   | If 1, the slot cannot participate in MAC commands.     |
| 5     | LimitedUse     | 0x0020   | If 1, slot usage is rate-limited by Counter0.          |
| 6     | EncRead        | 0x0040   | If 1, Read commands must be encrypted. (n/a, IsSecret=1) |
| 7     | IsSecret       | 0x0080   | If 1, slot value never readable in cleartext.          |
| 8     | GenKey         | 0x0100   | Allow GenKey to generate / regenerate the key.         |
| 9     | PrivWrite      | 0x0200   | Allow PrivWrite to import an encrypted key.            |
| 10-11 | Reserved       | 0x0C00   | Reserved, kept at 0.                                   |
| 12-15 | WriteConfig    | 0xF000   | See WriteConfig values below.                          |

#### SlotConfig layout for generic data slots

| Bit   | Field          | Mask     | Meaning                                                |
|-------|----------------|----------|--------------------------------------------------------|
| 0-3   | ReadKey        | 0x000F   | Slot index of the key used for encrypted reads.        |
| 4     | NoMac          | 0x0010   | If 1, slot cannot participate in MAC commands.         |
| 5     | LimitedUse     | 0x0020   | If 1, slot usage is rate-limited by Counter0.          |
| 6     | EncRead        | 0x0040   | If 1, Read commands must be encrypted.                 |
| 7     | IsSecret       | 0x0080   | If 1, slot value never readable in cleartext.          |
| 8-11  | WriteKey       | 0x0F00   | Slot index of the key used for encrypted writes.       |
| 12-15 | WriteConfig    | 0xF000   | See WriteConfig values below.                          |

#### WriteConfig values (bits 12-15)

| Value | Name                  | Behaviour for non-ECC data slots                                                                  |
|-------|-----------------------|----------------------------------------------------------------------------------------------------|
| `0x0` | Always                | Write allowed at any time before the data zone is locked. After data lock, Write is forbidden.     |
| `0x1` | PubInvalid            | Specialized for public-key validation; not used here.                                              |
| `0x2` | Never                 | Write in cleartext never allowed. (Encrypted PrivWrite for ECC slots still controlled by bit 9.)   |
| `0x4` | Always_then_Encrypt   | Cleartext Write allowed before data lock. After data lock, Write must be encrypted via WriteKey.   |
| `0x6` | Encrypt               | Cleartext Write never allowed. Encrypted Write via WriteKey always allowed.                        |
| `0x8` | Never                 | Never write. Functionally identical to `0x2` for most purposes.                                    |
| `0xC` | Never_then_Encrypt    | Cleartext never. After data lock, encrypted Write via WriteKey allowed.                            |

#### Per-slot values

##### Slots 0 and 1: ECC P-256, GenKey-only

Maximum security profile. The private key is generated on-chip and never
leaves it. Import is forbidden.

| Field        | Value | Reason                                                          |
|--------------|-------|-----------------------------------------------------------------|
| ExtSig       | 1     | We need to sign external messages with this key.                |
| IntSig       | 0     | Internal sign mode is unused.                                   |
| ECDH         | 0     | No ECDH operations in this project.                             |
| WriteECDH    | 0     | n/a.                                                            |
| NoMac        | 0     | We do not use MAC commands on these slots, but leaving 0.       |
| LimitedUse   | 0     | No use counter on Sign operations.                              |
| EncRead      | 0     | n/a, IsSecret=1.                                                |
| IsSecret     | 1     | The private key must never be readable.                         |
| GenKey       | 1     | The user must be able to regenerate the key for rotation.       |
| PrivWrite    | 0     | Import of an external private key is forbidden.                 |
| WriteConfig  | 0x2   | Cleartext Write is never permitted.                             |

Word value = `0b 0010 0001 1000 0001` = `0x2181`. Bytes little-endian: `81 21`.

##### Slots 2, 3, 4, 7, 9 through 15: ECC P-256, GenKey or encrypted import

User-flexible profile. The slot can be filled either by on-chip GenKey or
by importing an external key via PrivWrite (encrypted with the I/O
Protection Key in slot 8).

Same as the GenKey-only profile except:

| Field        | Value | Reason                                                          |
|--------------|-------|-----------------------------------------------------------------|
| PrivWrite    | 1     | Encrypted import of an external key is permitted.               |

Word value = `0b 0010 0011 1000 0001` = `0x2381`. Bytes little-endian: `81 23`.

##### Slots 5 and 6: data slots for PIN and PUK hashes

| Field        | Value | Reason                                                          |
|--------------|-------|-----------------------------------------------------------------|
| ReadKey      | 8     | Set to slot 8 for consistency (unused because EncRead=0).       |
| NoMac        | 0     | The CheckMac command must be allowed (used to verify the PIN).  |
| LimitedUse   | 1     | The chip rate-limits usage via Counter0 (slot 5) or Counter1 (slot 6). See below. |
| EncRead      | 0     | We never read the slot value back. The hash is verified by CheckMac, not Read. |
| IsSecret     | 1     | The hash must never leave the chip in cleartext.                |
| WriteKey     | 8     | When encrypted writes happen post-lock, the I/O Protection Key in slot 8 is used. |
| WriteConfig  | 0x4   | Always_then_Encrypt: cleartext Write before data lock (provisioning), encrypted Write after. |

Word value = `0b 0100 1000 1010 1000` = `0x48A8`. Bytes little-endian: `A8 48`.

**Important: LimitedUse and counter assignment.**
The ATECC608B has two monotonic counters (Counter0, Counter1). Both are
21-bit monotonic counters that, once the data zone is locked, can only
increment, never decrement, never reset.

The configuration uses `LimitedUse = 1` on slots 5 and 6. The
chip's convention is that **slot 5 is bound to Counter0 and slot 6 is bound
to Counter1**: a CheckMac on slot N that detects a mismatch atomically
increments Counter(N mod 2). When the counter reaches a project-defined
threshold (5 for PIN, 10 for PUK), the slot becomes hardware-locked.

The firmware uses a "batch refresh" pattern to give users a fresh batch of
N tries after each successful verification: it issues extra Counter
increment commands to bring the counter to the next multiple of N, so the
"tries remaining" metric (computed as `N - (counter mod N)`) resets to N.
The cost is N counter ticks per successful verification cycle.

With a 21-bit counter (about 2 million ticks) and 10 ticks consumed per
PUK reset cycle, the chip provides approximately 200 000 PUK reset cycles
over its life. Beyond that, the PUK feature becomes unavailable; the chip
remains usable as long as the user knows their PIN.

##### Slot 8: I/O Protection master key

The slot 8 master key is written once during provisioning, in cleartext,
while the data zone is still unlocked. Once the data zone is locked, this
slot becomes irrevocably immutable.

The key in slot 8 is used by the firmware to:

- Encrypt writes into slots 5 and 6 after the data zone has been locked
  (PIN and PUK updates).
- Encrypt imports of external ECC private keys into slots 2-4, 7, 9-15
  via PrivWrite.

The slot 8 master key is **32 random bytes drawn from the chip's RNG**
at provisioning time and never persisted anywhere on the device. The
host CLI returns the cleartext value once, in the response to
`ProvisionIoKey`, and the operator is responsible for backing it up.
The firmware holds no copy and cannot regenerate it.

This places the entire trust on the chip's RNG (DRBG-NIST SP 800-90A) at
provisioning time, with the firmware acting as a transparent conduit.
The advantage is that no long-term secret is baked into the firmware
binary: an attacker dumping flash gains nothing beyond the source code.

| Field        | Value | Reason                                                          |
|--------------|-------|-----------------------------------------------------------------|
| ReadKey      | 0     | Unused (EncRead=0, IsSecret=1).                                 |
| NoMac        | 0     | This slot acts as a key for MAC operations, so MAC must be allowed. |
| LimitedUse   | 0     | No use counter on this slot.                                    |
| EncRead      | 0     | Reads forbidden anyway by IsSecret=1.                           |
| IsSecret     | 1     | The master key must never leave the chip.                       |
| WriteKey     | 0     | Unused.                                                         |
| WriteConfig  | 0x0   | Always: cleartext Write allowed before data lock, never after.  |

Word value = `0b 0000 0000 1000 0000` = `0x0080`. Bytes little-endian: `80 00`.

### Bytes 52-67: Counter0 and Counter1 initial values

The ATECC608B encodes each monotonic counter in 8 bytes as a redundant
structure: two 16-bit linear ("lin") halves plus two 16-bit binary
("bin") halves. The chip decodes that storage to a single `u32` count
when queried via the `Counter` command. The factory initialization for
`count = 0` is:

```
Bytes 52-59 (Counter0): FF FF FF FF 00 00 00 00
Bytes 60-67 (Counter1): FF FF FF FF 00 00 00 00
```

**Writing `0xFF` to all 8 bytes does NOT mean "count = 0".** It leaves
the `bin_a` half at `0xFFFF`, which the chip interprets as a count near
the hardware ceiling (specifically `bin_a * 32 = 2_097_120`). A chip
config-locked with that stray initialization comes out of the lock with
~31 increments remaining out of `2^21 - 1` lifetime, effectively
ruining the chip for any production use.

The correct encoding for any target counter value is implemented in
`tools/config-generator/src/counter_encoding.rs`, mirroring
`calib_write_config_counter` from CryptoAuthLib
(`lib/calib/calib_basic.c`).

### Bytes 68-83: feature configuration (mostly unused)

| Offset | Field                 | Value     | Reason                                          |
|--------|-----------------------|-----------|-------------------------------------------------|
| 68     | UseLock               | `0x00`    | UseLock feature not used.                       |
| 69     | VolatileKeyPermission | `0x00`    | VolatileKey feature not used.                   |
| 70-71  | SecureBoot            | `0x00 00` | Secure boot not used.                           |
| 72     | KdflvLoc              | `0x00`    | KDF input vector unused.                        |
| 73-74  | KdflvStr              | `0x00 00` | KDF input vector seed unused.                   |
| 75-83  | Reserved3 (9 bytes)   | `0x00`    | Reserved, must stay 0.                          |

### Bytes 84-87

| Offset | Field         | Value  | Reason                                                    |
|--------|---------------|--------|-----------------------------------------------------------|
| 84     | UserExtra     | `0x00` | One-time write field, kept at default.                    |
| 85     | UserExtraAdd  | `0x00` | One-time write field, kept at default.                    |
| 86     | LockValue     | `0x55` | Data zone unlocked at provisioning. Chip writes `0x00` upon Lock(data).   |
| 87     | LockConfig    | `0x55` | Config zone unlocked at provisioning. Chip writes `0x00` upon Lock(config). |

The provisioning firmware writes `0x55` here. The chip itself will overwrite
these bytes to `0x00` when the corresponding Lock command is issued.

### Bytes 88-89: SlotLocked

A 16-bit bitmap, one bit per slot. **Bit value 1 means unlocked.** All slots
start unlocked at provisioning.

```
Value = 0xFFFF
Bytes little-endian: FF FF
```

The chip clears individual bits when `Lock(mode = slot, slot = N)` is issued.

### Bytes 90-91: ChipOptions

| Bit   | Field          | Mask     | Value | Meaning                                          |
|-------|----------------|----------|-------|--------------------------------------------------|
| 0     | POST_EN        | 0x0001   | 0     | Power-on self test disabled (faster wake).       |
| 1     | IO_PROT_EN     | 0x0002   | 0     | I/O Protection globally disabled.                |
| 2     | KDF_AES_EN     | 0x0004   | 0     | KDF/AES disabled.                                |
| 3-7   | Reserved       | 0x00F8   | 0     | Reserved.                                        |
| 8-9   | ECDH_PROT      | 0x0300   | 0     | ECDH protection unused.                          |
| 10-11 | KDF_PROT       | 0x0C00   | 0     | KDF protection unused.                           |
| 12-15 | IO_PROT_KEY    | 0xF000   | 0     | Irrelevant because IO_PROT_EN = 0.               |

```
Value = 0x0000
Bytes little-endian: 00 00
```

The mini-HSM does not use global I/O Protection because the firmware
encrypts writes case-by-case via the per-slot `WriteConfig=Encrypt` rules.
This keeps the protocol surface small and the firmware logic simple while
preserving end-to-end confidentiality on the writes that matter (PIN, PUK
updates).

### Bytes 92-95: X509format

Four bytes, one per `X509id` value (0 through 3). The mini-HSM does not
use X.509 signature formatting, so all four bytes are zero.

```
Value = 0x00 0x00 0x00 0x00
```

### Bytes 96-127: KeyConfig[16]

Each KeyConfig is 2 bytes, encoded as a little-endian u16 word.

#### KeyConfig layout

| Bit   | Field         | Mask     | Meaning                                                                |
|-------|---------------|----------|------------------------------------------------------------------------|
| 0     | Private       | 0x0001   | If 1, slot holds an ECC private key.                                   |
| 1     | PubInfo       | 0x0002   | For ECC: if 1, GenKey mode=public can compute the matching pubkey. For data slots: chip-specific KDF protection bit. |
| 2-4   | KeyType       | 0x001C   | Type of content. See table below.                                      |
| 5     | Lockable      | 0x0020   | If 1, the slot can be individually locked with Lock(mode=slot).        |
| 6     | ReqRandom     | 0x0040   | If 1, every use of the slot requires a fresh chip-side random nonce.   |
| 7     | ReqAuth       | 0x0080   | If 1, every use of the slot requires a successful CheckMac on AuthKey first. |
| 8-11  | AuthKey       | 0x0F00   | Slot index whose hash must be verified by CheckMac before use.         |
| 12    | PersistDis    | 0x1000   | Disables persistent latch use, not relevant here.                      |
| 13    | Reserved      | 0x2000   | Reserved, kept at 0.                                                   |
| 14-15 | X509id        | 0xC000   | X.509 format identifier, kept at 0.                                    |

#### KeyType values

| Value | Name        | Meaning                                            |
|-------|-------------|----------------------------------------------------|
| 0     | B283        | NIST B-283 ECC key.                                |
| 1     | K283        | NIST K-283 ECC key.                                |
| 4     | P256        | NIST P-256 ECC key. The only ECC type we use.      |
| 6     | AES         | AES symmetric key.                                 |
| 7     | Data        | 32 bytes of generic data (HMAC key, secret hash).  |

Other values are reserved by Microchip.

#### Per-slot values

##### Slots 0 and 1: ECC P-256, Lockable=0

| Field        | Value | Reason                                                                |
|--------------|-------|-----------------------------------------------------------------------|
| Private      | 1     | ECC private key.                                                      |
| PubInfo      | 1     | The firmware must be able to read the corresponding public key via GenKey mode=public. |
| KeyType      | 4     | P-256.                                                                |
| Lockable     | 0     | Slot cannot be individually locked. We always want regeneration possible. |
| ReqRandom    | 0     | No random nonce required before every Sign.                            |
| ReqAuth      | 1     | Sign and GenKey require a successful CheckMac on the auth slot first.  |
| AuthKey      | 5     | The auth slot is slot 5 (PIN hash). Sign requires PIN validation.      |
| PersistDis   | 0     | n/a.                                                                  |
| Reserved     | 0     | Reserved.                                                             |
| X509id       | 0     | Not an X.509 key.                                                     |

Word value = `0b 0000 0101 1001 0011` = `0x0593`. Bytes little-endian: `93 05`.

##### Slots 2, 3, 4, 7, 9 through 15: ECC P-256, Lockable=1

Same as slots 0 and 1 except `Lockable = 1`, so the slot can be individually
frozen later with `Lock(mode=slot, slot=N)`. This is useful for slot 10 if
it later receives an attestation key, for example.

Word value = `0b 0000 0101 1011 0011` = `0x05B3`. Bytes little-endian: `B3 05`.

##### Slots 5, 6, and 8: generic data slots

| Field        | Value | Reason                                                                |
|--------------|-------|-----------------------------------------------------------------------|
| Private      | 0     | Not an ECC key.                                                       |
| PubInfo      | 0     | n/a.                                                                  |
| KeyType      | 7     | Generic 32-byte data slot.                                            |
| Lockable     | 0     | We do not want to permanently freeze the PIN/PUK or I/O master key slots. |
| ReqRandom    | 0     | n/a.                                                                  |
| ReqAuth      | 0     | These slots are used as authorization sources, so no chained auth.    |
| AuthKey      | 0     | n/a.                                                                  |
| PersistDis   | 0     | n/a.                                                                  |

Word value = `0b 0000 0000 0001 1100` = `0x001C`. Bytes little-endian: `1C 00`.

## Final byte map of the writable portion (16-127)

The generated configuration zone, displayed as 16-byte rows for visual
inspection:

```
Offset    Bytes (hexadecimal)
------    -----------------------------------------------
  16-31:  C0 00 00 00 81 21 81 21 81 23 81 23 81 23 A8 48
  32-47:  A8 48 81 23 80 00 81 23 81 23 81 23 81 23 81 23
  48-63:  81 23 81 23 FF FF FF FF FF FF FF FF FF FF FF FF
  64-79:  FF FF FF FF 00 00 00 00 00 00 00 00 00 00 00 00
  80-95:  00 00 00 00 00 00 55 55 FF FF 00 00 00 00 00 00
  96-111: 93 05 93 05 B3 05 B3 05 B3 05 1C 00 1C 00 B3 05
 112-127: 1C 00 B3 05 B3 05 B3 05 B3 05 B3 05 B3 05 B3 05
```

The CRC-16 (polynomial 0x8005, init 0x0000, MSB first, little-endian
output, as defined in `crates/atecc608b/src/crc.rs`) of bytes 16-127 is:

```
CRC-16 (writable portion) = 0xC92D
```

This is the value the **`config-generator` blob** must produce. It is a
sanity check that the generator stays in sync with this specification,
not the value passed to `LockConfigZone`. The CRC the chip verifies at
lock time is computed over the **full 128 bytes**, including the per-chip
factory area, and is produced live by the host CLI from a fresh
`ReadConfigZone`.

## Verification procedure

Before any Lock command is issued, the operator must run the following
manual verification:

1. Run `tools/config-generator` to produce `config_zone.bin` (a 128-byte
   file). The byte map above must match the file bytes 16-127. The first
   16 bytes of the file are placeholders for the read-only factory area;
   the chip ignores them on Write.
2. Issue `WriteConfig(blob)` via the host CLI. The firmware writes the
   blob to the chip's configuration zone using the `Write` command in 4-byte
   chunks (the only chunk size that works pre-lock; 32-byte chunks become
   available only after the configuration zone is locked).
3. Issue `ReadConfig()` to read back the full 128 bytes from the chip.
4. Compare bit-by-bit. Bytes 0-15 will be the factory values; bytes 16-127
   must match the generated blob exactly.
5. Compute the CRC-16 of the read-back bytes 16-127. It must equal
   `0xC92D`. This is what `config-generator` would have produced from
   scratch; matching this value confirms the chip carries the intended
   configuration.
6. Only then, the operator may invoke `lock-config-DANGEROUS`. The CLI
   reads the chip's configuration once more, recomputes the CRC over the
   full 128 bytes (factory area included), shows it in the
   double-confirmation prompt, and forwards it to the chip. The chip
   recomputes the CRC one last time and refuses if the configuration has
   drifted between the read and the lock.

## Provisioning workflow (post config-lock)

Once the configuration zone is locked, the operator provisions the data
zone by invoking `hsm-host provision-token`. None of the steps below
lock the data zone. The data zone lock is a separate, optional,
irreversible operation.

The orchestrator issues these opcodes in order:

1. `ProvisionInitialPin`. The firmware computes
   `SHA-256("0000" || pin_salt)` on the chip and writes the result into
   slot 5 (allowed by `SlotConfig[5].WriteConfig = Always_then_Encrypt`
   while the data zone is still unlocked).
2. `ProvisionInitialPuk`. The firmware draws 8 ASCII digits from the
   chip RNG, computes `SHA-256(puk || puk_salt)` on the chip, writes the
   hash into slot 6, and returns the cleartext PUK in the response.
3. `ProvisionIoKey`. The firmware draws 32 random bytes from the chip
   RNG, writes them into slot 8 (allowed by
   `SlotConfig[8].WriteConfig = Always`), and returns the cleartext key
   in the response.
4. `GenKey(slot=0)` to materialise the primary identity key. The
   resulting public key is returned and recorded in the secrets file.

The host CLI persists the PUK, the I/O master key, the chip serial, and
the slot-0 public key to a JSON secrets file. These are the only copies
of the PUK and the I/O master key: the firmware retains no recoverable
form of either.

The data zone lock is then optional. Locking it freezes the slot 8
master key and prevents any further cleartext writes anywhere on the
chip. After data lock, PIN and PUK updates go through the encrypted-write
path (`SlotConfig WriteConfig = Always_then_Encrypt` becomes encrypted-only).

It is recommended to defer the data zone lock until late in development to
keep the option of reprovisioning.
